# ADR 0012: Triple-review as an external bash script invoking headless `claude -p`

## Status

Accepted

## Context

PR #134 attempted to implement `triple-review` as a Claude Code skill that orchestrates three pre-PR reviewers (`/pr-review-toolkit:review-pr`, `/security-review`, `/codex:adversarial-review`) from inside an active Claude Code session. Dogfooding surfaced structural blockers:

- `/codex:adversarial-review` and its sibling commands set `disable-model-invocation: true`. When orchestrated via the `Skill` tool, they are rejected.
- `/security-review` ends its prompt with "Your final reply must contain the markdown report and nothing else." When invoked via the `Skill` tool from a parent skill, this terminal rendering directive ends the parent turn and prevents further steps (e.g. aggregation) from running.
- The PR #134 implementation referenced a non-existent `BashOutput` tool and tried to find the Codex task via `TaskList`, which is racy with concurrent tasks.
- Three reviewers could not actually run in parallel from inside a session — the Skill tool serializes by turn.

The structural conclusion: **in-session orchestration is the wrong shape for this problem**. Each `claude -p` headless invocation is a fresh user-originated session where:

- `disable-model-invocation: true` does not apply (verified empirically with `claude -p "/codex:status"`).
- Sub-session rendering directives cannot terminate the parent (the parent is a bash script, not a Claude turn).
- True OS-level parallelism is trivial via bash `&` + `wait`.

Separately, the prior memory note claimed `/security-review` hardcodes `main...HEAD`. Binary inspection of Claude Code 2.1.116 disproves the hardcode claim — the system prompt is scope-agnostic ("the changes on this branch", "this PR") and the Claude Code runtime exposes a `getDefaultBranch` resolver that handles `origin/HEAD` → `origin/main`/`origin/master` fallback. `/security-review` relies on the LLM at runtime to invoke `gh pr view` / `git` commands to determine scope; its prompt does not hardcode `main`. Default-branch resolution correctness for non-`main` repos is therefore probabilistic rather than deterministic — verified empirically on this author's setup but not guaranteed by code inspection alone.

Binary inspection performed against Claude Code 2.1.116.

## Decision

Implement `triple-review` as an external bash script at `~/.local/bin/triple-review`, managed via chezmoi as `dot_local/bin/executable_triple-review`. The extensionless name matches the user-facing invocation (`triple-review`, not `triple-review.sh`); chezmoi's `executable_` prefix sets mode `0755`. The script runs outside any Claude Code session and is invoked from the terminal.

- **Entrypoint**: zero-argument command `triple-review`. No PR number, no `--base` flag, no working-tree mode. All three reviewers auto-detect scope.
- **Base resolution for the Codex `--base` flag**: `gh pr view --json baseRefName` on the current branch first. Fail-closed on unexpected `gh` errors (auth expiry, rate limit, network): aborting rather than falling back prevents the Codex leg from reviewing a different diff than the other reviewers (who auto-detect scope via their own `gh`/`git` calls), which would produce a summary mixing findings across diffs without any visible indication. Only when `gh` positively confirms no PR exists (stderr matches `no pull requests found`), fall back to `git symbolic-ref refs/remotes/origin/HEAD` with a warning about possible scope divergence. Abort if that fallback also fails.
- **Parallel execution**: three reviewers in the background, then `wait`.
  - `claude -p "/pr-review-toolkit:review-pr"` → `pr.md`
  - `claude -p "/security-review"` → `sec.md`
  - `node $CODEX_COMPANION adversarial-review --base <base> --scope branch --model gpt-5.4` → `adv.md` (stdout) + `adv.err` (stderr / `[codex]` progress lines). Direct codex-companion invocation, NOT `claude -p "/codex:adversarial-review --wait …"`. See "Workaround — direct codex-companion invocation" and the model pin note under Known limitation below.
- **Error handling**: hybrid. If at least one reviewer succeeds, continue; emit a `<FAILED>` marker in the aggregation prompt for any failed reviewer. If all three fail, abort with `exit 1` and surface the log paths.
- **Aggregation**: a fourth `claude -p` call wraps the three outputs in unique-delimiter-bounded data blocks (`===BEGIN_TRIPLEREV_<entropy>_<REVIEWER>===` / `===END_...===`) and asks for a priority-based summary (`対応必須` / `要検討` / `対応不要` / `矛盾`) with `file:line` evidence. The `<entropy>` token is `date +%s%N` on GNU coreutils (Linux) or seconds + PID + 8 bytes from `/dev/urandom` on BSD `date` (macOS, where `%N` is returned as the literal string `N`). The per-run delimiter prevents reviewer output from spoofing block boundaries (e.g. an embedded `</review_pr>` no longer breaks out of the data block). The prompt also explicitly instructs the aggregator to treat embedded imperatives as data, never as directives. Aggregator stdout and stderr are redirected to separate files (`summary.md`, `summary.err`) so post-rc=0 stderr warnings cannot contaminate the user-facing summary tail. Reference to official docs via `WebFetch`/`WebSearch` is explicitly permitted in the prompt.
- **Output**: stdout contains all four sections (three raw + aggregated summary). Intermediate files are written to `$TMPDIR/triple-review-<XXXXXX>/` (random suffix via `mktemp -d`); the path is reported just before the handoff section. `$TMPDIR` is cleaned by macOS automatically; on Linux, cleanup depends on `tmpfiles.d` or manual removal.
- **Auto-handoff**: after printing the summary, when stdout is a TTY the script `exec`s into interactive `claude` with `summary.md` content + a follow-up prompt as the initial message — dropping the user directly into a Claude Code session to act on findings instead of landing back at the shell prompt. When stdout is not a TTY (piped/redirected), the auto-handoff is skipped and the summary file path is printed as a fallback hint, so automation doesn't jump into a broken interactive UI.
- **Progress UX**: a short banner before the background jobs (`Running 3 reviewers in parallel...` plus the resolved base branch, work directory, and a `tail -f` tip) and an `All reviewers done. Generating summary...` line after `wait`. While reviewers run, a background monitor subshell polls the three PIDs every 10 seconds and emits a status line of the form `[Nm SSs] PR:run SEC:done ADV:run` whenever state changes *or* every 2 minutes as a heartbeat — so silence never exceeds the heartbeat window, and the user can see at a glance which reviewer is holding up the pipeline.
- **Timeouts**: none. Reviewers typically run 2-10 minutes; interruption is via Ctrl+C.
- **Signal handling**: a `trap` on `EXIT`/`INT`/`TERM` snapshots the full descendant tree of each tracked `claude -p` PID via recursive `pgrep -P` (capturing MCP servers, sub-agents, `gh`/`git` invocations, and deeper descendants), sends `SIGTERM` to the entire collected set, waits a `sleep 0.2` (200ms) grace, then escalates to `SIGKILL` for anything still alive. The snapshot is taken BEFORE `SIGTERM` because grandchildren reparent to PID 1 (`launchd` on macOS / `systemd` on Linux) once their parent exits, and `pgrep -P <old-parent>` can no longer locate them. Process-group-based signalling (`kill -TERM -$pgid`) is intentionally rejected: bash scripts run without job control, so backgrounded `claude -p &` shares the script's PGID and `kill -- -$pgid` would SIGTERM the script itself; `setsid(1)` to isolate each child into its own group is Linux-only (macOS lacks the command). Recursive `pgrep -P` is therefore the portable route.
- **Sleep inhibition**: On startup (after argument validation but before `check_prerequisites`), the script auto-wraps itself via `exec caffeinate -i -s` (macOS) or `exec systemd-inhibit --what=idle:sleep --mode=block --why='triple-review in progress'` (Linux with logind). On macOS, `-i` inhibits idle sleep on both AC and battery (primary goal), and `-s` adds system-sleep inhibition that applies only on AC power — per `man caffeinate(8)`, the `-s` assertion "is valid only when system is running on AC power". On Linux, `--what=idle:sleep` drops `shutdown` from systemd-inhibit's default (`idle:sleep:shutdown`, per `man systemd-inhibit`) so the tool never blocks an operator-initiated shutdown; `handle-lid-switch` is not in the default and is intentionally left unset so lid-close still suspends if the user intends that. Re-entry is guarded by `TRIPLE_REVIEW_SLEEP_INHIBITED=1`, which the parent exports immediately before `exec`. The self-exec argv is `"${wrapper[@]}" "${BASH:-bash}" "${BASH_SOURCE[0]}" "$@"` — explicitly routing re-entry through the running bash interpreter rather than `execve`-ing the script file. The repository stores the script as mode 0644 (chezmoi sets 0755 only on apply), so any source-tree invocation — `bash dot_local/bin/executable_triple-review`, the bats wrapper, local editor-driven runs — would fail with rc=126 if the inhibitor tried to `execve($0)` directly. Production (installed 0755) would work either way, but the bash-interpreter shape also decouples the wrap path from any future mode-bit regression. Three guards precede the Linux emit: (1) `has_systemd_inhibit` (binary on PATH), (2) `systemd_is_pid1` (`/run/systemd/system` exists), (3) `systemd_inhibitor_reachable` — an acquisition probe that invokes `systemd-inhibit --what=idle:sleep --mode=block --why='triple-review probe' /bin/true`, which exercises the same polkit action the production wrap will hit while releasing the lock within milliseconds (`/bin/true` returns immediately). The guards are named for what they literally test: (2) only confirms systemd is PID 1, which is necessary but not sufficient for logind reachability or polkit acquisition permission; (3) is the actual acquisition test (see "Probe vs. acquisition" below). The third guard exists because (2) alone can be true in WSL2 without systemd-genie, rootless podman, and sudo-stripped environments where logind is not reachable over dbus; in those cases exec would succeed, bash would be replaced by systemd-inhibit, and the child would exit before handing off to triple-review, leaving the user with no review at all. Silent fall-through on any guard miss — wrapping is best-effort; the script itself must always run. `shopt -s execfail` additionally lets the parent recover if the wrapper `exec` itself fails (e.g. a race where the inhibitor binary disappears between probe and exec); because `set -Eeuo pipefail` would otherwise exit the shell on a failed exec before the `||` branch fires, the call is bracketed with `set +e` / `set -e` so the warn + continue fallback actually runs. Laptop battery clamshell sleep remains uncontrollable by user-space; neither `caffeinate` nor `systemd-inhibit` can prevent hardware-level lid-switch suspension.

**Probe vs. acquisition (guard 3)**: An earlier implementation used `systemd-inhibit --list` for guard (3) because it round-trips dbus to logind. That signal is necessary but not sufficient: restrictive polkit policy (observed on hardened multi-user hosts and some distro defaults that gate `org.freedesktop.login1.inhibit-*` actions behind `auth_admin`) lets `--list` succeed yet denies the actual sleep-inhibit acquisition. With a reachability-only probe, the wrap decision said "go", `exec systemd-inhibit ...` replaced bash with the wrapper, polkit rejected the lock, the wrapper exited non-zero, and triple-review never ran — the same silent-disappearance failure mode the WSL2 / rootless guard was added to prevent. The acquisition probe exercises the exact polkit action the production wrap will trigger and is bounded to milliseconds by `/bin/true`'s immediate return. Both probe and wrap deliberately spell out `--mode=block` (currently the systemd-inhibit default) so a future default change cannot silently reintroduce a probe-vs-wrap mismatch. Issue #158 captures the original report and reproduction steps on Linux 6.17 + restrictive desktop polkit policy.

## Consequences

- **Positive**: Real OS-level parallelism via `claude -p &` for the PR/SEC reviewers + direct `node codex-companion.mjs` for the ADV reviewer. The original design used `claude -p` uniformly for all three to "survive plugin upgrades," but the headless slash-command dispatch path turned out to hang for `/codex:adversarial-review`, forcing the direct-CLI approach for ADV (see "Workaround — direct codex-companion invocation" below).
- **Positive**: Real OS-level parallelism cuts wall-clock time roughly to the slowest reviewer instead of the sum of three.
- **Positive**: Hybrid error handling preserves partial results — a single reviewer failure no longer discards the 2-10 minutes of work the others already did.
- **Positive**: Priority-based summary with four Japanese sections (`対応必須` / `要検討` / `対応不要` / `矛盾`) matches the manual review workflow the author used before automation, surfacing "which should I actually fix" directly instead of requiring a second triage pass. All reviewer-facing output is in Japanese per the repository's global `CLAUDE.md` convention.
- **Negative**: Cannot be invoked from within an active Claude Code session — requires dropping to the terminal. This is by design; in-session orchestration was the source of every PR #134 blocker.
- **Negative**: Four `claude -p` calls per run (three reviewers + one aggregation) cost more than a single session turn. Each reviewer runs for many turns against full codebase context (especially `/pr-review-toolkit:review-pr`, which internally spawns several sub-agents). Expected several USD per run, dominated by total token consumption rather than per-call overhead. Actual cost should be measured post-implementation.
- **Risk**: `disable-model-invocation: true` behavior in `claude -p` mode (relevant to `/pr-review-toolkit:review-pr` and `/security-review`) relies on current Claude Code semantics (verified on 2.1.116). If Anthropic tightens the flag to also block headless mode in a future release, those legs break and would require equivalent direct-script invocations. The Codex leg already uses direct codex-companion invocation (see "Workaround — direct codex-companion invocation" below) so it is not exposed to this risk.
- **Risk**: Three parallel `claude -p` processes consume Anthropic API quota three-fold simultaneously. For individual-tier users this is within normal bursts; heavy usage could hit per-minute limits.
- **Known limitation — scope alignment**: Only the Codex leg receives an explicit `--base`. `/pr-review-toolkit:review-pr` and `/security-review` do not accept a base flag at the slash-command level and auto-detect scope at runtime via their own `gh pr view` / `git` calls. When a PR is associated with the current branch, all three reviewers converge on the same `baseRefName` (high confidence). When no PR exists, the Codex leg falls back to `origin/HEAD` while the other two rely on their internal heuristics — which may resolve a different base if the repo is misconfigured. In the no-PR case the script emits a warning to stderr so the user can cross-check the three diffs before trusting the aggregated summary. Opening a draft PR before running `triple-review` is the cheapest way to guarantee scope alignment.
- **Known limitation — prompt injection into aggregator**: Reviewer outputs are LLM-generated text derived from an untrusted diff and are inserted verbatim into the aggregation prompt. Defense-in-depth in four layers: (1) each block is bounded by a per-run unique delimiter `===BEGIN_TRIPLEREV_<entropy>_<REVIEWER>===` / `===END_...===` (entropy from `date +%s%N` on Linux, or seconds + PID + 8 urandom bytes on macOS where BSD `date` lacks `%N`) that reviewer LLMs cannot predict, so embedded static-looking tags like `</review_pr>` cannot spoof block boundaries; (2) the prompt explicitly tells the aggregator that only the unique-delimiter markers are valid boundaries and any other tag-like strings are inert data; (3) the prompt also instructs that embedded imperative text ("ignore previous", "システムメッセージ", etc.) is data and must not be followed as directives; (4) the interactive handoff passes the summary via `claude -- "<prompt>"` with a `--` separator so that summaries beginning with `-` or a commander.js subcommand name (`agents`, `auth`, `mcp`, `plugin`, etc.) cannot redirect CLI dispatch away from the positional prompt path. The aggregator output is read by the user rather than executed, so the blast radius is integrity of the summary (not code execution or data exfiltration).
- **Known limitation — partial-failure visibility**: When 1-2 reviewers fail, `triple-review` continues with partial results (hybrid error handling, above) and exits 0. To prevent silent coverage loss, the script prepends a blockquote header to the `## Summary` section listing which reviewers failed whenever `fail_count > 0`. This keeps the "preserve partial work" benefit while making degraded coverage impossible to miss on stdout.
- **Positive — sleep robustness**: Auto-wrapping with the OS sleep inhibitor prevents the "76-minute wall-clock with ~47 minutes suspended" scenario observed in PR #135 testing, where macOS system sleep froze all three `claude -p` children and their progress monitor. Users no longer need to remember `caffeinate` manually.
- **Known limitation — laptop clamshell on battery**: No user-space mitigation exists for hardware-level lid-switch suspension on battery. Closing the lid while running on battery will still suspend the job. `print_usage` documents the AC-power workaround for users who hit this.
- **Known limitation — codex plugin sandbox requires `~/.claude/plugins/data/codex-openai-codex` write allowance**: The codex plugin's app-server persists per-workspace state under `~/.claude/plugins/data/codex-openai-codex/state/<slug>/`. Claude Code's process sandbox (`enabled: true`) blocks writes outside `sandbox.filesystem.allowWrite` — without that entry, `codex-companion` fails with `EROFS` (Linux) / `EPERM` (macOS) on the first state write. In interactive `/codex:adversarial-review`, Claude Code detects the failure and auto-retries with the sandbox disabled; in headless `claude -p` the auto-recovery does not trigger and the failure surfaces as a broker-handshake hang. The dotfiles `private_dot_claude/settings.json` adds `~/.claude/plugins/data/codex-openai-codex` (the codex-specific subdirectory only — not the data root) to the allowlist so any sandboxed Claude Code path that touches codex-companion (interactive `/codex:adversarial-review`, `/codex:rescue`, `/codex:status`, etc.) can persist state. The narrower scope follows ADR 0001's least-privilege principle: writes by other plugins under `~/.claude/plugins/data/` remain blocked unless explicitly allowed. (`triple-review` itself runs outside any Claude Code session and is therefore unaffected by Claude Code's sandbox; this allowlist entry is for the other invocation paths.) The path holds plugin runtime state only, not user secrets, so the security impact of permitting writes is minimal.
- **Known limitation — codex plugin model default broken on codex-cli 0.125.0 + plugin 1.0.4**: The default `gpt-5.5` configured at the top of `~/.codex/config.toml` returns a 400 from OpenAI when invoked through the codex plugin's adversarial-review path. The plugin's structured-output handler swallows the error rather than propagating it as a `turn/error` notification, so `captureTurn` awaits `turn/completed` indefinitely (upstream `openai/codex-plugin-cc#183`) and the broker stays "busy" — a second `codex app-server` then spawns via `withAppServer`'s `disableBroker` fallback (`scripts/lib/codex.mjs:607-636`), leaving two stuck workers per stuck job. End result: `adv.md` stays empty / placeholder; the aggregator receives no Codex content. Mitigation: pin `--model gpt-5.4` on the ADV invocation. Revert when upstream `openai/codex-plugin-cc#270` is resolved. Tracked locally as #157.
- **Workaround — direct codex-companion invocation (bypassing `claude -p` for ADV)**: Even with the model pin and the sandbox allowance above, `claude -p "/codex:adversarial-review --wait …"` (the original headless path) still hangs at `phase: starting` for 8+ minutes. Empirically, invoking the underlying script directly — `node ~/.claude/plugins/cache/openai-codex/codex/<version>/scripts/codex-companion.mjs adversarial-review --base <base> --scope branch --model gpt-5.4` — runs the same companion code in foreground and completes normally with real review output. The hang is therefore in the slash-command dispatch layer of headless `claude -p`, not in the companion script itself. The companion path is resolved dynamically at startup (`resolve_codex_companion`): the cache layout is `<root>/<version>/scripts/codex-companion.mjs` with no `latest` symlink, so any hard-coded version string eventually breaks under cache pruning or a new install. The resolver globs the versioned dirs and `sort -V`-picks the highest, so a leftover older version alongside a newer one does not silently bind to the older contract. Pinning to a specific cache path was rejected outright: the cache is not a stable interface (`~/.claude/plugins/cache/`) and is subject to background prune. To prevent silent contract drift between resolution and the workaround revert, `extraKnownMarketplaces.openai-codex.autoUpdate` is set to `false` in `private_dot_claude/settings.json` — version bumps are gated on a manual `/plugin update`. stderr is split into `adv.err` so `adv.md` stays as pure review markdown (the bare CLI emits `[codex] Starting Codex task thread …` style progress lines on stderr). Plan: revert to `claude -p "/codex:adversarial-review …"` once the slash-command dispatch hang is resolved upstream and uniformity is desirable again; re-enable `autoUpdate` at the same time.
- **Known side effect — orphan broker after auto-handoff session quit (resolved in #163)**: Originally, the bare-CLI `node codex-companion.mjs adversarial-review …` invocation did not tear down its `app-server-broker.mjs` on completion — broker lifecycle is normally tied to the Claude Code session that launched the slash command, but the bare-CLI path has no such session. When `triple-review`'s auto-handoff then `exec`ed into interactive `claude --`, the codex plugin's `SessionStart` hook adopted the orphan broker. On user quit, the `SessionEnd` hook attempted a graceful shutdown via the broker socket; if the broker was unresponsive, Claude Code's hook timeout fired (~60s) and the user saw `SessionEnd hook [...session-lifecycle-hook.mjs SessionEnd] failed: Hook cancelled`. The broker process and its `/var/folders/.../cxc-*/{broker.sock,broker.pid}` were left behind (process re-parented to PID 1). Resolved by adding a sibling `triple-review-broker-cleanup` Node helper that drives the plugin's `broker-lifecycle.mjs` API (`sendBrokerShutdown` → `teardownBrokerSession` → `clearBrokerSession`) synchronously before the auto-handoff `exec`. The helper takes a pre-snapshot of `broker.json` existence at startup and only tears down brokers we created — pre-existing brokers (concurrent same-repo Claude Code sessions) are left alone, satisfying the PID-targeted cleanup constraint. The plugin internal API (`loadBrokerSession`, `sendBrokerShutdown`, `teardownBrokerSession`, `clearBrokerSession` from `lib/broker-lifecycle.mjs`; `terminateProcessTree` from `lib/process.mjs`) is not a stable surface; `private_dot_claude/settings.json` pins `autoUpdate: false` for the openai-codex marketplace so manual `/plugin update` is the only thing that can change it. If the API surface changes incompatibly on a future update, the helper exits non-zero and triple-review emits a warn — the review run still succeeds, but the orphan-broker symptom returns until the helper is updated. The cleanup is also wired into `kill_children` (the EXIT/INT/TERM trap target) so Ctrl+C and aggregator-failure paths also clean up. Manual cleanup if the helper degrades silently: locate the temp dir via `ps -ef | grep app-server-broker.mjs` (the `--pid-file` argument exposes the path), read the PID from `<cxc-dir>/broker.pid`, then `kill <pid>` and `rm -rf <cxc-dir>`. Avoid `pkill -f 'app-server-broker.mjs'` — broad pattern matching also terminates brokers owned by concurrent Claude Code sessions or other repositories that legitimately have a broker running.
